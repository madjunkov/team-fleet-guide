Team-Facing Claude Agent Fleet

A practical guide to spinning up a fleet of team-facing Claude Code agents (Analytics, CMO, CFO, CPO, etc.) on a dedicated Mac mini host, separate from your personal machine.

Why a separate host

The personal-machine fleet works for one operator, but breaks for team-facing agents:

  • Resource contention. Each LLM session pins ~1–2 GB of RAM. Six personas plus per-user DM workers under each is 12+ sessions. A laptop running 12 tmux sessions of Claude Code is slow at the day job.
  • Sleep / lid-close kills the fleet. A team can't have their bots disappear when you close your laptop.
  • Personal config bleed. Your ~/.claude/settings.json, your model defaults, your ~/.zshrc — these are your workspace. The fleet host needs to be its own identity.
  • Audit clarity. Slack messages from "stojan's mac" vs "team-fleet host" tell different stories to compliance reviewers.

The cleanest pattern: a dedicated Mac mini (or Studio) wired to power + Ethernet, running the fleet 24/7. Your laptop stays free for the day job. You SSH into the fleet host (via Tailscale or VPN) to attach to sessions and observe.

The architecture

Each team-facing persona is a three-tier system:

┌───────────────────────┐ │ Slack workspace │ │ (or Telegram group) │ └─────────┬─────────────┘ │ events / DMs ▼ ┌───────────────────────────────────┐ │ Tier 1: Slack-side bot │ │ (slack-fork plugin holds socket) │ │ • Receives events │ │ • Writes to inbound-queue │ │ • Posts replies via chat.post │ └─────────┬─────────────────────────┘ │ queue file ▼ ┌───────────────────────────────────┐ │ Tier 2: Non-LLM router daemon │ │ (analytics-router.ts pattern) │ │ • Reads dm-users.json registry │ │ • Forwards to user's worker │ │ • Relays worker replies │ │ • Deterministic, can't go rogue │ └─────────┬─────────────────────────┘ │ local-handoff inbox ▼ ┌───────────────────────────────────┐ │ Tier 3: Per-user LLM worker │ │ (claude --continue per Slack ID) │ │ • The actual conversation │ │ • Has tools, knowledge access │ │ • One session per allowed user │ └───────────────────────────────────┘

Tiers 1 + 3 are the obvious bits. Tier 2 is the load-bearing part that beginners skip and then regret: without a non-LLM router between the Slack-side and the per-user workers, the LLM "router" keeps answering questions itself instead of forwarding — because it has the tools, and the prompt isn't strong enough to enforce the boundary.

Lesson from the field We initially built the analytics agent with an LLM as the router (since it already had mcp__slack__reply). It worked for one user, but as soon as we wired in per-user workers, the LLM router kept choosing the path of least resistance — calling mcp__slack__reply directly instead of forwarding to the user's worker. Prompt-only enforcement failed; we had to replace it with a non-LLM daemon. Don't skip Tier 2.

Choosing a Mac mini

Recommended starting spec:

ComponentRecommendedNotes
ModelMac mini M4 (base or Pro)M-series chips run multiple Claude sessions cleanly. Avoid Intel minis.
RAM16 GB minimum, 24+ GB idealEach LLM session is ~1–2 GB. Fleet of 6 personas + 5–10 workers each = comfortable in 24 GB.
Storage512 GB SSD minimumClaude session JSONLs accumulate. Log rotation helps but headroom is cheap.
NetworkEthernet preferredWi-Fi works but adds reconnect latency.
Cost$600 used / $799 new (base)Pro is $1399. Base M4 is enough for a 6-persona fleet.

The host doesn't need a monitor — once configured, you SSH in. Tailscale (free for personal use) is the simplest way to reach it from anywhere without exposing it to the public internet.

First-boot prerequisites

On the fresh Mac mini:

# 1. Create the fleet-host user account (a dedicated user, not your personal one)
#    Recommended: create user "fleet" in System Settings → Users & Groups

# 2. SSH on:
sudo systemsetup -setremotelogin on

# 3. Tailscale for remote access:
brew install --cask tailscale
# Then sign in via the Tailscale menu bar app

# 4. Core toolchain:
brew install tmux jq git
curl -fsSL https://bun.sh/install | bash

# 5. Claude Code CLI:
curl -fsSL https://claude.ai/install.sh | bash
claude /login   # authenticate this host's user with Anthropic

# 6. GitHub CLI (optional but recommended):
brew install gh
gh auth login
Tip — pin macOS to never sleep System Settings → Energy → "Prevent computer from sleeping when display is off." Otherwise your fleet quietly disappears at 2am. Also disable auto-update reboots: sudo softwareupdate --schedule off.

The fleet state directory layout

Everything fleet-related lives under ~/.claude/:

~/.claude/
├── channels/                  # one dir per bot identity / channel
│   ├── slack-analytics/       # the analytics persona's Slack state
│   │   ├── .env               # SLACK_BOT_TOKEN + SLACK_APP_TOKEN (0600)
│   │   ├── access.json        # allowed channels, per-user grants
│   │   ├── dm-users.json      # per-Slack-user worker registry
│   │   ├── inbound-queue/     # Tier 1 → Tier 2 message queue
│   │   └── bot.pid            # liveness heartbeat
│   ├── slack-cmo/             # next persona, same shape
│   ├── slack-cfo/             # ...
│   └── local-handoff/         # inter-agent message bus
│       └── inbox/
│           ├── analytics/
│           ├── analytics-dm-worker-U6D3CRN85/
│           └── ...
├── plugins/local/             # custom MCP plugins (slack-fork, local-handoff)
├── prompts/                   # per-persona system prompts
│   ├── analytics.txt          # public-channel LLM session
│   ├── analytics-dm-worker.txt # per-user DM worker
│   ├── cmo.txt
│   └── ...
├── bin/                       # daemons + CLIs
│   ├── analytics-router.ts    # non-LLM Tier 2 daemon
│   ├── handoff.sh             # inter-agent relay
│   └── slack-dm-cleanup.ts    # one-offs
└── settings.json              # global Claude Code config

The pattern is regular: each persona gets its own ~/.claude/channels/slack-<persona>/ with the same five files inside. The plugin code is shared.

Anatomy of a single persona

A complete persona — say "CMO" — consists of these pieces, each takeable from the analytics template:

  1. Slack app: A registered Slack app at api.slack.com/apps with Socket Mode enabled, im:history + im:write + chat:write + reactions:write scopes, and an app-level token for socket mode. Provides the bot identity, like @BVG Agent - CMO.
  2. Channel state dir: ~/.claude/channels/slack-cmo/ with .env (the bot's tokens, 0600), access.json (which Slack channels and users it can talk to), and dm-users.json (which Slack users get their own DM worker session).
  3. Router daemon: A copy of analytics-router.ts parameterized for "cmo." Reads ~/.claude/channels/slack-cmo/inbound-queue/, dispatches to cmo-dm-worker-<user_id> sessions. LaunchAgent com.bvg.cmo-router with KeepAlive.
  4. Public-channel LLM session: A tmux session named cmo-public running claude --append-system-prompt-file=~/.claude/prompts/cmo.txt. Handles inbound from the persona's public Slack channel.
  5. Per-user DM workers: One tmux session per allowed user, named cmo-dm-worker-<SLACK_USER_ID>. Each running claude --continue --append-system-prompt-file=~/.claude/prompts/cmo-dm-worker.txt with its own cwd at ~/cmo-dm-workers/user-<UID>/.
  6. Resume aliases in ~/.zshrc: claude-cmo-resume brings up the public session + reconciles workers. claude-cmo-stop tears it down.

Slack app bootstrap (script-driven)

Slack's App Manifest API lets you create new apps programmatically. The recipe:

Get an app-configuration token

This is a one-time setup that requires workspace admin rights. At api.slack.com/authentication/config-tokens, generate an app-configuration token (starts with xoxe.xoxp-). This token can create + modify other Slack apps.

Sensitive token The app-config token is workspace-admin-equivalent. Store it in ~/.claude/.bootstrap-token with mode 0600. Don't commit it.

Bootstrap script

Create ~/.claude/bin/persona-bootstrap.sh:

#!/usr/bin/env bash
# Usage: persona-bootstrap.sh <persona_name> <display_name>
# Example: persona-bootstrap.sh cmo "BVG Agent - CMO"
set -euo pipefail

PERSONA="$1"
DISPLAY="$2"
CONFIG_TOKEN=$(cat ~/.claude/.bootstrap-token)

MANIFEST=$(cat <<EOF
{
  "display_information": {
    "name": "${DISPLAY}",
    "description": "BVG ${PERSONA} agent — team-facing AI assistant",
    "background_color": "#1a1a1a"
  },
  "features": {
    "bot_user": {
      "display_name": "${DISPLAY}",
      "always_online": true
    },
    "app_home": {
      "messages_tab_enabled": true,
      "messages_tab_read_only_enabled": false
    }
  },
  "oauth_config": {
    "scopes": {
      "bot": [
        "chat:write",
        "channels:history",
        "im:history",
        "im:read",
        "im:write",
        "reactions:write",
        "users:read"
      ]
    }
  },
  "settings": {
    "event_subscriptions": {
      "bot_events": ["message.im", "app_mention"]
    },
    "socket_mode_enabled": true
  }
}
EOF
)

RESULT=$(curl -s -X POST "https://slack.com/api/apps.manifest.create" \
  -H "Authorization: Bearer ${CONFIG_TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{\"manifest\": $(echo "$MANIFEST" | jq -Rs .)}")

APP_ID=$(echo "$RESULT" | jq -r .app_id)
echo "Created app_id: $APP_ID"
echo "Next: install to workspace at https://api.slack.com/apps/${APP_ID}/install-on-team"
echo "Then copy bot token + app token into ~/.claude/channels/slack-${PERSONA}/.env"

Run: bash ~/.claude/bin/persona-bootstrap.sh cmo "BVG Agent - CMO" → outputs an app_id. Then click the install link (Slack requires one human confirmation for workspace install, by design).

The non-LLM router daemon pattern

The router (Tier 2) is the architectural choice that makes a team-facing fleet sane. It's a small TypeScript daemon, ~600 lines, that:

  • Watches a single inbound-queue directory via fs.watch + 2s poll backstop
  • For each inbound: looks up the registry, decides "forward to worker X" vs "reject with refusal"
  • For each worker reply: posts to the right Slack DM channel, threaded correctly
  • Never invokes an LLM. Has no Anthropic API calls. Cannot go rogue.

Key implementation notes from production:

  • FIFO queue per worker. Workers reply via handoff.sh which doesn't preserve the inbound's audit_id. So you can't key the reply-context map on audit_id. Maintain a per-worker FIFO of pending forward contexts; each worker reply pops the oldest. Workers process events serially, so FIFO is a sound proxy for "this reply is for the earliest unfulfilled forward."
  • Content-hash dedup. Workers (especially Sonnet) sometimes call handoff.sh two or three times "to verify it worked." Dedup against the body content, not audit_id, within a 60-second window.
  • Defensive threading. Slack DMs expect threaded replies. If the operator's inbound was a top-level message, use its ts as the new thread_ts for the bot's reply (creates a new thread under that message). If the inbound was already a thread reply, inherit the parent thread_ts.
  • LaunchAgent with KeepAlive: true and ThrottleInterval: 30. If the daemon dies, macOS restarts it within 30 seconds. The 30-second floor prevents crashloops from spamming logs.

Permissions and security model

Two boundaries to think about:

Per-Slack-user access

The dm-users.json registry is the allowlist. Users not in it get a polite refusal:

{
  "users": {
    "U0286CPMP": { "name": "Albert", "model": "claude-sonnet-4-6" },
    "U6DRS8QG6": { "name": "Ryan",   "model": "claude-sonnet-4-6" }
  }
}

The router daemon checks this on every inbound. To add a user: edit the file, no restart needed (daemon re-reads on each event).

Per-action grants on the public channel

The persona's public Slack channel uses a separate access.json with per-user action grants. The operator gets freeform (trusted passthrough), everyone else gets report (untrusted submission, surfaces as a contained framed event the LLM can log but won't act on without operator confirmation).

{
  "allowedChannels": ["C0B57B5RG6N"],
  "users": {
    "U6D3CRN85": ["freeform"]
  },
  "defaultActions": ["report"],
  "implicitAction": "report"
}

What the LLM cannot do autonomously

Two operator-only invariants enforced by both prompt and tooling:

  1. No autonomous infra changes. Agents never swap tokens, edit .env files, modify access.json, or change MCP bindings. They diagnose problems and present options; the operator decides.
  2. Destructive actions need trusted-channel confirmation. Anything irreversible (DB writes, deletes, force-pushes, fleet config changes) requested via Slack must be confirmed by the operator on a separate trusted surface — usually Telegram DM to a dedicated ops bot.

Operations: starting, stopping, observing

Bring the whole fleet up after reboot

# A single command via ~/.claude/bin/fleet-startup.sh
~/.claude/bin/fleet-startup.sh

# Which does:
#   pm2 resurrect                   # restores any pm2-managed processes
#   claude-analytics-resume         # public channel LLM session
#   claude-analytics-dm-restore-workers   # all per-user workers
#   claude-cmo-resume + workers
#   ...
#   launchctl loads the LaunchAgents (already auto-load on reboot)

Observe live state

# All tmux sessions
tmux ls

# Attach to a specific session
tmux attach -t analytics-public
tmux attach -t analytics-dm-worker-U6D3CRN85

# Detach (don't kill) with: Ctrl-b d

# Tail the router daemon log
tail -f ~/.claude/analytics-router.log

# Check LaunchAgents
launchctl list | grep com.bvg

SSH from your laptop

# With Tailscale set up on both ends:
ssh fleet@<fleet-host-tailnet-name>

# Then once on the host:
tmux attach -t analytics-public

Common failure modes

SymptomLikely causeFix
Bot gets 👀 on a DM but never replies slack-fork plugin running inert (missing token, wrong state dir) Check SLACK_STATE_DIR env in the launching alias. Verify .env contains SLACK_BOT_TOKEN and SLACK_APP_TOKEN.
Bot DM gets multiple identical replies Worker LLM calling handoff.sh multiple times "to verify" Tighten worker prompt with explicit "one inbound = one reply" rule + daemon-side content-hash dedup.
Reply lands at top-level instead of threaded Daemon's reply-context lookup failing (most likely keyed on audit_id which is "0" under --audit-via=none) Use FIFO per-worker matching, not audit_id keying.
Plugin process alive but no Slack TCP connection Plugin started in inert mode (silent stderr-only warning) lsof -p <pid> | grep TCP — if no ESTABLISHED https connection, the plugin loaded without tokens. Check env.
"channel_not_found" errors on chat.postMessage Bot lacks im:write scope OR not invited to channel Slack app config → OAuth & Permissions → add scope → reinstall.
Worker tmux fills with dialog prompts First-boot folder-trust / dev-channels / MCP-server dialogs need dismissing Run a single shared umbrella daemon that polls all tmux sessions and dismisses known dialogs. tmux send-keys with the right key sequences.

Recipe: adding a new persona in 30 minutes

Once the host is set up, adding "CMO" is mechanical:

  1. (5 min) Bootstrap the Slack app: bash ~/.claude/bin/persona-bootstrap.sh cmo "BVG Agent - CMO" → click the install link → copy tokens.
  2. (2 min) Create state dir: mkdir -p ~/.claude/channels/slack-cmo && cd $_ && touch .env access.json dm-users.json && chmod 600 * Populate .env with the tokens.
  3. (3 min) Configure access: Edit access.json with the public channel ID + operator grant. Edit dm-users.json with the team members who can DM this persona.
  4. (5 min) Write the prompt: Copy ~/.claude/prompts/analytics.txt to cmo.txt + analytics-dm-worker.txt to cmo-dm-worker.txt. Edit the persona-specific scope sections.
  5. (5 min) Add resume aliases: Copy the analytics block in ~/.zshrc as a starting point for claude-cmo-resume + claude-cmo-dm-restore-workers. sed-replace "analytics" → "cmo." Mind the model pin: --model claude-opus-4-8 for the public-channel LLM.
  6. (5 min) Spawn the router daemon: Copy ~/.claude/bin/analytics-router.ts to cmo-router.ts, replace the state-dir constant. Copy the LaunchAgent plist to com.bvg.cmo-router.plist. launchctl load it.
  7. (5 min) Test end-to-end: DM @BVG Agent - CMO from an allowlisted user. Verify daemon log shows FORWARDED → worker pane shows local-handoff inbound → daemon log shows RELAYED → reply lands in Slack.

If step 7 fails, the diagnostics dance is well-trodden: lsof -p on the plugin (confirm TCP connection), tail -f the router log (confirm queue write), check tmux capture-pane on the worker (confirm dialog dismissal + event arrival).

End-state Once a persona is running smoothly, the operator forgets about it. Team members DM @BVG Agent - CMO like they'd DM a colleague; the bot routes to their per-user worker; the worker responds. The Mac mini quietly serves the team. No babysitting.
