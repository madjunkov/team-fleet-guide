Rollout phases

From "we have an idea" to "the team is using it every day" in four phases. Each phase ends with a checkpoint where you can decide whether to continue.

Phase 1

Foundation

Provision the dedicated host. Install runtime (bun, tmux, Claude Code). Set up Tailscale for remote SSH access. Configure macOS to never sleep.

~2 hours · one-time
Phase 2

First persona

Ship one persona end-to-end (typically Analytics, since it has the clearest scope). Register the Slack app, wire up the router daemon, prove DM routing works for one user.

~4 hours · one-time
Phase 3

Expand allowlist

Add team members to the first persona's allowlist. Each user gets their own conversation context via per-user worker sessions. Tune the prompt based on real questions.

~30 min · ongoing
Phase 4

Add personas

Once the analytics flow proves itself, spin up CMO, CFO, CPO, etc. Each follows the same recipe in ~30 minutes — Slack app bootstrap, state dir, router daemon, prompts.

~30 min per persona

How a single persona works

Three tiers, each doing one job. Information flows top-to-bottom on inbound, bottom-to-top on reply.

Tier 1 · Slack connection

The bot's socket to Slack

Holds the persistent WebSocket to Slack. Receives every event (DMs, mentions, reactions). Posts replies via Slack's API. Doesn't decide anything on its own — just routes inbound to Tier 2 and outbound to Slack.

↓ inbound    ↑ reply
Tier 2 · Non-LLM router

Deterministic dispatcher

A small TypeScript daemon, no LLM. For each inbound message:

  • Checks the per-user allowlist
  • Authorized → forwards to that user's dedicated worker
  • Not authorized → posts "Not authorized yet" + an eyes reaction

For each worker reply: posts to the right DM channel, threaded correctly. Can't go rogue because it has no model to second-guess the routing.

↓ forward    ↑ reply
Tier 3 · Per-user workers

The actual conversation

One Claude Code session per allowlisted user. This is where the work happens — answering questions, running queries, drafting documents. Each user has their own conversation context (preserved across days), their own scratch directory, their own model choice. Cleanly isolated from other users' conversations.

Why the middle tier? Tempting to skip Tier 2 and let an LLM handle routing directly — it already has the tools, just tell it "forward to the right worker." This doesn't work in practice: the LLM keeps choosing the path of least resistance and answers questions itself instead of forwarding. The non-LLM router is the load-bearing architectural choice. Why a dedicated host

The fleet doesn't run on your laptop. Four reasons:

ConcernWhy this matters
Always-on Team members can't have their bots disappear when you close your laptop or commute. A dedicated host plugged into power + Ethernet is the bare minimum.
Resource contention Each Claude session uses ~1–2 GB of RAM. Six personas × ~5 active workers each is 30+ sessions. Your laptop slows down; the fleet host is sized for it.
Identity clarity Bots running from "stojan's mac" look personal. Bots running from "BVG Agent Host" read as company infrastructure for compliance + audit.
Config separation Your shell history, your Claude Code defaults, your environment — these are your workspace. The fleet host should be its own thing.

What the user sees

Just Slack. Someone on the marketing team DMs @BVG Agent - CMO with a question. The bot responds in their DM, threaded under the question. Conversations persist across days — the user can ask follow-ups weeks later and the agent remembers context.

From the team's perspective: it's an always-available colleague who knows their function's data and tools. No new app to install, no new login, no integration burden. Just Slack.

What the operator does

After the initial setup (Phase 1 + Phase 2), the operator's day-to-day workload is light:

  • Add new team members to a persona's allowlist (edit one JSON file, no restart needed)
  • Tune prompts based on real feedback ("the bot is too verbose," "needs to default to summarizing first")
  • Add new personas as new functions ask for one (~30 min per persona)
  • Monitor via Telegram DM to a dedicated ops bot — the operator's audit + control channel separate from team-facing surfaces

The fleet doesn't need babysitting. macOS LaunchAgents restart anything that crashes. Log rotation handles disk growth. The operator gets pinged on Telegram only when something needs decision-making.

Security model

Two invariants enforced by code, not by trust:

  1. No autonomous infrastructure changes. Agents diagnose problems and present options. They never swap tokens, edit access lists, or modify their own configs. The operator decides every infra-level change.
  2. Destructive actions require trusted-channel confirmation. If someone in a public Slack channel asks the bot to delete data or push code, the bot logs the request and asks the operator via Telegram before doing anything. The operator's Telegram DM is the only "trusted" surface for irreversible actions.

Per-Slack-user allowlists prevent randoms in the workspace from DMing the bot to fish for information. The allowlist is a single JSON file the operator edits; no UI, no approval flow.
